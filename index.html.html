🎯 INTERVIEW STRATEGY
BEFORE THE INTERVIEW
Research the company's tech stack (LinkedIn job description, GitHub repos, engineering blog). Tailor your answers to use tools they actually use.
Prepare 3 real stories from your current job using STAR method: Situation → Task → Action → Result. Numbers matter: "reduced deployment time from 2 hours to 10 minutes."
Set up your home lab and practice on real infra. Saying "I set this up in my home lab" beats saying "I read about it." Interviewers can tell the difference.
Do a mock interview out loud, not just in your head. Your brain thinks you know it until your mouth has to produce it. Record yourself and review.
Have your GitHub profile with real projects. 3 quality repos beats 30 toy projects. Include a README with architecture diagram.
DURING TECHNICAL QUESTIONS
When asked "How does X work?" — always go 2 levels deep. "A Pod is a wrapper around containers" is level 1. "The kubelet watches the API server, gets Pod spec, calls containerd via CRI to create container, sets up cgroups/namespaces" is level 2. Level 2 gets you hired.
If you don't know something exactly — say "I haven't used X in depth, but based on my understanding of similar tools…" Never bluff. Interviewers know immediately and will lose trust in everything else you said.
Think out loud when solving problems. "I'd first check pod logs, then describe the pod for events, then check the node, then check if resource limits are hit…" — process matters as much as answer.
Draw architecture diagrams when describing systems, even if just verbally. "So we have a VPC with public and private subnets, the ALB sits in public, it routes to pods in private subnets via NodePort…"
COMMON MISTAKES IN DEVOPS INTERVIEWS
❌ Saying "We use Kubernetes" without explaining WHY or WHAT problem it solved.
❌ Confusing Container vs Image vs Pod vs Deployment — these are different things. Know exact definitions.
❌ Not knowing kubectl troubleshooting steps (kubectl describe, logs, exec, events).
❌ Describing CI/CD without mentioning rollback strategy, failure handling, or monitoring.
❌ Giving only AWS answers when Azure/GCP also asked — know conceptual mapping between clouds.
❌ Treating "tell me about yourself" as casual. This is your brand pitch. Prepare it: name, current role, key achievement, why DevOps, what you're looking for — 90 seconds max.

Docker CMD vs ENTRYPOINT — The Restaurant Analogy
ENTRYPOINT = The chef (fixed — can't change what kind of restaurant this is)
CMD = The default order (can override when customer says something else)
Together: ENTRYPOINT ["python", "app.py"] CMD ["--port", "8080"]
When you run: docker run image --port 9090 → ENTRYPOINT stays, CMD is replaced
ENTRYPOINT = fixed identity, CMD = default argument
Kubernetes Services — The Hotel Analogy
ClusterIP = Internal phone extension (only inside hotel — pod to pod)
NodePort = Hotel's main number (outside can reach, via specific port 30000-32767)
LoadBalancer = Cloud's front desk with external IP (cloud provider creates real load balancer)
ExternalName = Forwarding phone number (maps to external DNS name, no proxying)
C-N-L-E: Confined, Narrow door, Lobby, External
AWS IAM — "UPAG" Framework
UPAG
Users = Individual humans (avoid long-lived keys; use roles instead)
Policies = JSON permission documents (attach to anything)
Access = via Groups (assign users to groups, attach policies to groups)
Grant temporarily = via Roles (EC2 instance roles, Lambda roles, cross-account access)
Golden rule: Never hardcode AWS keys in code. Always use IAM roles.
Security Groups vs NACLs — STAPLES vs FILTER
Security Groups (SG):
STAPLES
Stateful · Track connections · Attached to instances · Permit only (whitelist only) · Lowest level granularity · Evaluates all rules · Supports inbound/outbound separately
NACLs:
FILTER
Firewalls for subnets · Inbound and outbound · Lower rule number wins · Traceless (stateless) · Explicit deny possible · Rule order matters
Terraform State — The Blueprint vs Building Analogy
HCL code = What you WANT the building to look like (desired state)
State file = What ACTUALLY exists (current state — terraform.tfstate)
terraform plan = Comparing blueprint vs current building — shows differences
terraform apply = Construction crew builds what's missing or changes what's wrong
Never edit state file manually. Use terraform state mv/rm/import commands.
CI/CD Deployment Strategies — The Traffic Light Analogy
Recreate = 🔴 Red light — stop everything, then go. Full downtime.
Rolling = 🟡 Yellow light — gradually replace old with new. Brief partial downtime.
Blue-Green = 🟢 Green light — run both, flip traffic switch instantly. Zero downtime, needs 2x infra.
Canary = 🟢 with safety net — send 5% of traffic to new, watch metrics, ramp up.
R-R-B-C: Risky-Reasonable-Bold-Careful
MLOps Pipeline — "DTESMD" Framework
DTESMD
Data ingestion → Transformation/preprocessing → Experiment/training → Selection/evaluation → Model registry → Deployment/serving
Plus monitoring loop: track data drift, model drift, prediction quality → trigger retraining
Tools map: MLflow (E+S+M) | Kubeflow (D+T+E) | KServe/Seldon (D) | Evidently (monitoring)
Prometheus Metric Types — "CGHS"
CGHS
Counter = only increases (requests total, errors total) — use rate() for per-second
Gauge = can go up/down (memory usage, active connections, queue depth)
Histogram = buckets for distribution (request duration) — use histogram_quantile()
Summary = like histogram but quantiles calculated client-side — less flexible
Interview answer: "I'd use a Counter for total requests, Gauge for current connections, Histogram for latency p95/p99."
